Mòdul 4: Patrons de Comportament

Chain of Responsibility

Què és el Patró Chain of Responsibility?

El patró Chain of Responsibility és un patró de disseny de comportament que permet que un conjunt d'objectes gestionin una sol·licitud de manera seqüencial. Cada objecte de la cadena té l'oportunitat de processar la sol·licitud o passar-la al següent objecte de la cadena.

Objectius del Patró

  • Desacoblament: Separar l'emissor d'una sol·licitud dels seus receptors.
  • Flexibilitat: Permetre que múltiples objectes gestionin una sol·licitud sense especificar explícitament el receptor final.
  • Extensibilitat: Facilitar l'addició de nous gestors a la cadena sense modificar el codi existent.

Components del Patró

  1. Handler (Gestor): Defineix una interfície per gestionar les sol·licituds i, opcionalment, enllaçar amb el següent gestor de la cadena.
  2. ConcreteHandler (Gestor Concret): Implementa la interfície del gestor i gestiona les sol·licituds que pot processar. Si no pot processar la sol·licitud, la passa al següent gestor de la cadena.
  3. Client: Inicia la sol·licitud a un objecte de la cadena.

Diagrama UML

Client --> Handler --> ConcreteHandler1 --> ConcreteHandler2 --> ConcreteHandler3

Exemple de Codi

A continuació, es presenta un exemple en Python per il·lustrar el patró Chain of Responsibility:

class Handler:
    def __init__(self, successor=None):
        self._successor = successor

    def handle(self, request):
        raise NotImplementedError("Aquest mètode ha de ser implementat per subclasses")

class ConcreteHandler1(Handler):
    def handle(self, request):
        if 0 < request <= 10:
            print(f"ConcreteHandler1 gestiona la sol·licitud {request}")
        elif self._successor:
            self._successor.handle(request)

class ConcreteHandler2(Handler):
    def handle(self, request):
        if 10 < request <= 20:
            print(f"ConcreteHandler2 gestiona la sol·licitud {request}")
        elif self._successor:
            self._successor.handle(request)

class ConcreteHandler3(Handler):
    def handle(self, request):
        if 20 < request <= 30:
            print(f"ConcreteHandler3 gestiona la sol·licitud {request}")
        elif self._successor:
            self._successor.handle(request)

# Configuració de la cadena de responsabilitat
handler1 = ConcreteHandler1()
handler2 = ConcreteHandler2(handler1)
handler3 = ConcreteHandler3(handler2)

# Client que envia sol·licituds
requests = [5, 14, 22, 18, 3, 27]

for request in requests:
    handler3.handle(request)

Explicació del Codi

  1. Handler: Defineix la interfície base amb el mètode handle i un successor opcional.
  2. ConcreteHandler1, ConcreteHandler2, ConcreteHandler3: Implementen el mètode handle per gestionar sol·licituds dins d'un rang específic. Si no poden gestionar la sol·licitud, la passen al següent gestor.
  3. Configuració de la cadena: Es crea una cadena de gestors on ConcreteHandler3 passa les sol·licituds a ConcreteHandler2, que al seu torn les passa a ConcreteHandler1 si no les pot gestionar.
  4. Client: Envia diverses sol·licituds que són gestionades per la cadena de gestors.

Avantatges i Desavantatges

Avantatges:

  • Desacoblament: L'emissor de la sol·licitud no necessita conèixer el receptor final.
  • Flexibilitat: És fàcil afegir o canviar gestors a la cadena.
  • Responsabilitat distribuïda: Cada gestor només gestiona les sol·licituds que pot processar.

Desavantatges:

  • Depuració complexa: Pot ser difícil seguir el flux de sol·licituds a través de la cadena.
  • Rendiment: Si la cadena és molt llarga, pot afectar el rendiment.

Conclusió

El patró Chain of Responsibility és una solució elegant per gestionar sol·licituds de manera flexible i desacoblada. Permet que múltiples objectes tinguin l'oportunitat de processar una sol·licitud, facilitant l'extensibilitat i la mantenibilitat del codi. Tot i això, cal tenir en compte la complexitat afegida en la depuració i el possible impacte en el rendiment.
